# hachi-position Moat and Wedge Guide

Moat does not need to be strong on day one. It must describe what compounds with continued use and why the position may remain useful after competitors copy surface features.

## Moat Candidates

- Workflow moat: the product becomes part of the user's repeated operating rhythm.
- Context moat: the product accumulates project, team, domain, or decision context.
- Data moat: repeated use creates structured evidence, traces, scores, or histories.
- Integration moat: value grows through connections to GitHub, docs, PRD, ADR, Wiki, Deck, or local files.
- Community moat: users contribute examples, templates, policies, or workflows.
- Local-first moat: sensitive data, offline use, transparency, or portability matters.
- Japanese-context moat: domestic workflows, language, documents, norms, or business practices matter.
- Process moat: the product encodes a decision or review process users trust.
- Taste moat: outputs reflect opinionated, high-quality judgment that generic tools do not reproduce.
- Agentable moat: outputs are structured so agents and humans can both act on them.

## Wedge Checklist

Define:

- First user
- First pain
- Workflow entry point
- First input
- First output
- First distribution form
- Why now
- Why not a platform feature
- What later expansion becomes possible

Prefer wedges that start as a Skill or CLI when the value is analysis, review, conversion, or local file transformation. Prefer GitHub Action or IDE extension only when the pain happens inside CI or coding flow.

## Weak Position Smells

- The target user is "everyone using AI".
- The product category is just "dashboard" or "AI assistant".
- The differentiation is only "better UX".
- The moat is only "we collect data" without explaining why data quality compounds.
- Vendor risk is dismissed without identifying a niche.
- The MVP requires integrations before the core value is proven.
