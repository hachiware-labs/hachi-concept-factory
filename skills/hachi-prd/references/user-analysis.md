# hachi-prd User Analysis

PRDs must describe users in operational detail. Do not treat target users as labels.

## Required User Analysis

```markdown
## User Analysis

### Primary User
- Role:
- Context:
- Current workflow:
- Technical level:
- Constraints:
- Success condition:
- Failure consequence:

### Secondary Users / Stakeholders
- Reader:
- Approver:
- Operator:
- Maintainer:

### User Journey
- Before:
- Trigger:
- During:
- After:
- Repeated use:
```

## Pain Point Format

```markdown
### Pain 1: ...
- Situation:
- Trigger:
- Current workaround:
- Why existing alternatives fail:
- Frequency:
- Severity:
- Emotional tone:
- Business / workflow impact:
- Willingness to pay or adopt:
- Evidence:
```

## Quality Bar

- Name who operates the product and who reads the output.
- Describe the user's environment, existing tools, constraints, and technical level.
- Include first-use and repeated-use behavior.
- Connect every major functional requirement to a pain point or use case.
- If user data is missing, infer conservatively and label the inference.
